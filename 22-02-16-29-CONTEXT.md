## 2. ARCHITECTURE DES FICHIERS & COMPOSANTS

Le routing est manuel (`useState` sur `activeTab`), tout est injecté depuis `App.jsx`.

```
/src
 ├── App.jsx               # Entrypoint. Gère le layout global (Sidebar, Topbar, Statusbar) et le state lifting `activeAgentId`, `activeTab`.
 ├── index.css             # Base CSS : grid overlay (tactical), noise SVG, animations custom pures (`animate-data-enter`, `pulse-fast`), `.tactical-panel`.
 ├── main.jsx              # DOM Mount
 │
 ├── /components
 │   ├── AgentsPanel.jsx   # Grille exhaustive des agents. Affiche la charge (System Load progress), tokens, statut et bouton "UPLINK".
 │   ├── CommsConsole.jsx  # Interface de chat isolée par agent (Map). Terminal (gauche) + Data Context Agent (droite).
 │   ├── IntelFeed.jsx     # Mini-sidebar (droite) : log stream (débrayable visuellement) pour les views autres que Comms/Intel.
 │   ├── IntelPanel.jsx    # Station approfondie : Full Log Stream filtrable (severity, sources) + Threat Matrix Tooltip-enriched + Source Breakdown.
 │   ├── OverviewPanel.jsx # Hub principal. 6 widgets statiques : TaskPipeline summary, CognitionLoad, ActiveDirectives, AgentTelemetry, ThreatHeatmap, SystemTimeline.
 │   ├── Sidebar.jsx       # Navigation tabs + Roster rapide des agents (points clignotants vert/rouge).
 │   ├── Statusbar.jsx     # Barre inférieure d'informations système techniques (Ground Truth).
 │   ├── TasksPanel.jsx    # Hub Kanban Dnd-kit complet avec DragOverlay (5 colonnes) + bascule sur Timeline View chronological.
 │   └── Topbar.jsx        # Horloge UTC et Santé globale du système (Ingestion, Cognition, Comms).
 │
 └── /hooks
     └── useSimulation.js  # CERVEAU DU MOCK. Gère un état global (système, tâches, agents, feed, map) via un setInterval massif crachant de la donnée aléatoire cohérente.
```

**A. `agents` / `agentStatus` (Array)**

## 5. DÉCISIONS TECHNIQUES MAJEURES

1. **State Management de la `CommsConsole.jsx` (Chat Isolation)** : L'historique des discussions n'est plus un array global mais une `Map(agentId -> Object[])`. Changer d'agent préserve hermétiquement les conversations. Les réponses asynchrones génèrent via proxy les messages d'accueil et les hooks. L'`activeAgentId` a dû être remonté (`Lifted State`) dans `App.jsx` pour permettre aux autres vues (comme `AgentsPanel`) de déclencher une redirection vers un agent cible (bouton `UPLINK`).
2. **Double State du Kanban (Dnd-Kit)** : `useSimulation` n'arrête pas de cracher de nouvelles références d'objets `tasks`. Pour éviter que le Dnd-kit n'explose lors du drag d'une task, `TasksPanel.jsx` synchronise et fige un état local `localTasks`. Toute manipulation de cartes Drag & Drop modifie `localTasks`. 
3. **Composants Puristes** : En dehors de `recharts` pour le graphe fluid de Cognition, l'entièreté des micro-animations (Pulse du `Uplink`, `data-enter` slide in de l'historique, strobe rouge d'alerte, barre de chargement) est faite en CSS Tailwind pour privilégier la performance devant les setInterval de la simulation.

## 6. BRANCHÉ vs SIMULÉ

**ABSOLUMENT TOUT EST SIMULÉ MOCKÉ.**
Le `useSimulation.js` tourne en fond via un hook effect qui invoque un `setInterval` à 1 seconde. Il exécute des algorithmes de type Chaîne de Markov pour incrémenter les stats et switcher aléatoirement l'état des agents pour faire "vivre" l'interface graphique. Personne ne parle en réseau. 

**Exemple d'interaction mockée cible (Uplink)** : `CommsConsole` initie de faux messages en utilisant `setTimeout(..., 1200ms)`.

## 7. POINTS D'EXTENSION : CONNECTER OPENCLAW / BACKEND

Pour relier cette interface vide à l'orchestrateur IA d'OpenClaw, vous modifierez ou supprimerez `useSimulation.js` et vous connecterez un layer de gestion asynchrone (ex: `WebSockets` ou `React Query`).

**Ce qui doit être fait :**
1. Créer une connexion à `wss://votre-backend/events`.
2. Convertir les objets entrants pour qu'ils matchent les structures définies dans la **section 4** (Format des paylods). 
3. **Kanban Backend** : Implémenter le dispatch API. Dans `TasksPanel.jsx`, la méthode `handleDragEnd` effectue un arrayMove visuel. Il va falloir injecter une requête `fetch('PATCH /tasks/:id')` avec le nouveau `overContainer` pour synchroniser le backend lorsque l'opérateur lâche une carte.
4. **Comms Uplink** : Dans `CommsConsole.jsx`, hook le `handleSubmit` pour émettre via `socket.emit('operator_message', { agentId: activeAgentId, msg: inputVal })` au lieu du faux `setTimeout`.

## 8. CE QU'IL RESTE À FAIRE (TODO & FEATURES INCOMPLÈTES)

1. L'export de log depuis `IntelPanel` copie bien textuellement dans le presse papier, mais il n'exécute aucun export fichier type `.csv` ou `.md`.
2. Le système n'est pas "Mobile-Friendly". Très focalisé Desktop/Large Screen (`xl:`). De gros problèmes d'affichage surgiront sur smartphone (Kanban écrasé, colonnes de chat compressées).
3. Le Drag&Drop de `TasksPanel` écrase et ignore les mises à jour en cours du serveur. Cela n'est pas un bug front, mais lors de l'intégration WebSocket, il faudra prévoir une stratégie pour éviter la disparition d'une carte si le serveur la met à jour au moment où l'utilisateur a la main dessus.
